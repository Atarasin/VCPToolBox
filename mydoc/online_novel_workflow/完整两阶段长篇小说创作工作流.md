## 二、第一阶段：前期设定产出

### 2.1 阶段目标

产出完整的小说前期设定文档，包括：
- 世界观设定（world_building.md）
- 人物设定（characters.md）
- 分卷大纲（volume_outline.md）
- 章节细纲（chapter_outline.md）

### 2.2 质量闭环机制：正反辩论

每层设定都通过**设计者 vs 挑刺者**的多轮辩论来保障质量。

```mermaid
flowchart TB
    subgraph DebateLoop["设计-辩论循环"]
        direction TB
        
        Start([开始]) --> PreCheck{预检}
        PreCheck -->|不通过| InputError[返回错误]
        PreCheck -->|通过| Designer[设计者生成方案]
        
        Designer --> Critic[挑刺者审核]
        Critic --> Score{评分}
        
        Score -->|>=85分| Pass[审核通过]
        Score -->|<85分| Feedback[收集反馈]
        
        Feedback --> Counter{迭代次数}
        Counter -->|<3次| Designer
        Counter -->|>=3次| MaxIter[达到上限]
        
        MaxIter --> Manual[转人工审核]
        Pass --> Archive[存档]
        Manual --> Archive
        
        Archive --> End([结束])
        InputError --> End
    end
```

### 2.4 单层的完整流程

以世界观设定为例：

```mermaid
sequenceDiagram
    actor User as 用户
    participant PC as 预检模块
    participant D as 设计者Agent
    participant C as 挑刺者Agent
    participant DB as 辩论控制器
    participant A as 存档模块
    
    User->>PC: 提交世界观需求
    PC->>PC: 检查输入完整性
    
    alt 预检不通过
        PC-->>User: 返回错误信息
    else 预检通过
        PC->>DB: 启动辩论循环
        
        loop 最多3轮辩论
            DB->>D: 生成世界观方案
            D-->>DB: 返回方案
            
            DB->>C: 审核方案
            C-->>DB: 返回评分和问题
            
            alt 评分>=85
                DB->>A: 存档
                A-->>User: 返回最终方案
            else 评分<85且未到3轮
                DB->>D: 反馈问题，继续迭代
            else 评分<85且已到3轮
                DB->>User: 转人工审核
            end
        end
    end
```

## 三、第二阶段：章节创作产出

### 3.1 阶段目标

基于前期设定，逐章创作小说正文，并通过质量闭环保障输出质量。

### 3.2 质量闭环机制：预检-创作-审核-回流

```mermaid
flowchart TB
    subgraph ChapterLoop["单章创作质量闭环"]
        direction TB
        
        Start([开始]) --> LoadContext[加载上下文]
        LoadContext --> PreCheck{预检}
        
        PreCheck -->|不通过| PreFail[预检失败]
        PreCheck -->|通过| Generate[深潜创作]
        
        Generate --> Review[质量审核]
        Review --> Decision{审核结果}
        
        Decision -->|通过| Archive[存档交付]
        Decision -->|不通过| Reflow{回流决策}
        
        Reflow -->|重试创作| Counter{计数检查}
        Reflow -->|人工介入| Manual[转人工]
        
        Counter -->|<3次| Generate
        Counter -->|>=3次| Manual
        
        Archive --> End([结束])
        Manual --> End
        PreFail --> End
    end
```

### 3.3 预检阶段

检查输入条件是否满足，避免无效创作。

```mermaid
flowchart LR
    subgraph PreCheck["预检检查项"]
        direction TB
        PC1[大纲完整性检查]
        PC2[依赖满足性检查]
        PC3[上下文充分性检查]
        PC4[人物状态检查]
        PC5[伏笔状态检查]
    end
    
    Input[输入] --> PC1
    PC1 --> PC2
    PC2 --> PC3
    PC3 --> PC4
    PC4 --> PC5
    PC5 --> Result{结果}
    
    Result -->|全部通过| Pass[进入创作]
    Result -->|任一失败| Fail[返回错误]
```

**预检检查项说明：**

| 检查项 | 检查内容 | 失败处理 |
|--------|----------|----------|
| 大纲完整性 | 章节细纲是否存在且完整 | 终止，返回错误 |
| 依赖满足性 | 前置章节是否已完成 | 等待或跳过 |
| 上下文充分性 | 角色状态、世界观信息是否加载 | 补充上下文 |
| 人物状态检查 | 人物当前状态是否符合设定 | 修正状态 |
| 伏笔状态检查 | 待回收伏笔是否已记录 | 补充记录 |

### 3.5 质量审核阶段

多维度审核生成的章节内容。

```mermaid
flowchart TB
    subgraph Review["质量审核"]
        direction TB
        
        Input[章节内容] --> ParallelReview["并行审核"]
        
        ParallelReview --> Consistency[一致性审核]
        ParallelReview --> Completeness[完整性审核]
        ParallelReview --> Quality[基础质量审核]
        
        Consistency --> ConsistencyResult{结果}
        Completeness --> CompletenessResult{结果}
        Quality --> QualityResult{结果}
        
        ConsistencyResult --> Aggregate[结果聚合]
        CompletenessResult --> Aggregate
        QualityResult --> Aggregate
        
        Aggregate --> FinalResult{最终判定}
        FinalResult -->|全部通过| Pass[通过]
        FinalResult -->|存在问题| Fail[不通过]
    end
```

**审核维度说明：**

| 审核维度 | 检查内容 | 通过标准 |
|----------|----------|----------|
| **一致性审核** | 人物名、地点名是否与设定一致 | 0处严重不一致 |
| **人设一致性** | 人物言行是否符合设定 | 0处严重偏离 |
| **情节一致性** | 情节是否与大纲一致 | 大纲覆盖率≥90% |
| **完整性审核** | 是否覆盖大纲要求的情节点 | 情节点覆盖率≥95% |
| **字数达标** | 字数是否在目标范围 | 达标率90%-110% |
| **基础质量** | 语法、错别字、标点 | 无明显错误 |

### 3.6 回流决策阶段

根据审核结果决定回流路径。

```mermaid
flowchart TB
    subgraph Reflow["回流决策"]
        direction TB
        
        Input[审核结果] --> Classify[问题分级]
        
        Classify --> Critical[严重问题]
        Classify --> Major[重要问题]
        Classify --> Minor[轻微问题]
        
        Critical --> CheckCounter1{计数检查}
        Major --> CheckCounter2{计数检查}
        Minor --> PassMinor[通过并记录]
        
        CheckCounter1 -->|<3次| Replan[晨会重规划]
        CheckCounter1 -->|>=3次| Manual1[人工介入]
        
        CheckCounter2 -->|<3次| Redo[深潜重创作]
        CheckCounter2 -->|>=3次| Manual2[人工介入]
        
        Replan --> Output1[返回预检]
        Redo --> Output2[返回创作]
        PassMinor --> Output3[通过]
        Manual1 --> Output4[人工]
        Manual2 --> Output4
    end
```

**回流路径说明：**

| 问题级别 | 定义 | 回流路径 | 计数器 |
|----------|------|----------|--------|
| **严重** | 影响理解/设定冲突/人设崩塌 | 晨会重规划 | 回流计数+1 |
| **重要** | 影响体验/逻辑漏洞 | 深潜重创作 | 迭代计数+1 |
| **轻微** | 可优化/不影响主线 | 通过并记录建议 | 不计数 |

### 3.7 完整单章创作流程

```mermaid
sequenceDiagram
    actor Controller as 工作流控制器
    participant PreCheck as 预检模块
    participant Context as 上下文构建
    participant LLM as LLM接口
    participant Review as 质量审核
    participant Reflow as 回流决策
    participant Archive as 存档模块
    
    Controller->>PreCheck: 执行预检
    PreCheck-->>Controller: 预检结果
    
    alt 预检不通过
        Controller->>Controller: 记录错误
    else 预检通过
        loop 最多3次迭代
            Controller->>Context: 构建上下文
            Context-->>Controller: 上下文包
            
            Controller->>LLM: 生成章节
            LLM-->>Controller: 章节正文
            
            Controller->>Review: 质量审核
            Review-->>Controller: 审核结果
            
            alt 审核通过
                Controller->>Archive: 存档
                Archive-->>Controller: 完成
            else 审核不通过且未到3次
                Controller->>Reflow: 回流决策
                Reflow-->>Controller: 回流路径
                Note over Controller: 继续循环
            else 审核不通过且已到3次
                Controller->>Controller: 转人工介入
            end
        end
    end
```

## 九、关键设计决策

### 9.1 为什么采用正反辩论机制？

| 方案 | 优点 | 缺点 |
|------|------|------|
| 单Agent生成 | 简单快速 | 质量不可控 |
| **正反辩论** | **质量有保障，问题前置发现** | **实现复杂，token消耗高** |
| 人工审核 | 质量最高 | 效率最低 |

**决策**：采用正反辩论，在质量和效率之间取得平衡。

### 9.2 为什么最多3轮迭代？

| 迭代次数 | 质量提升 | 成本 | 边际效益 |
|----------|----------|------|----------|
| 1轮 | 60% | 低 | - |
| **3轮** | **85%** | **中** | **最优** |
| 5轮 | 90% | 高 | 递减 |

**决策**：3轮迭代可以捕获80%以上的问题，边际效益最优。

### 9.3 为什么通过阈值设为85分？

- 90分+：过于严格，容易导致无限迭代
- 80分-：过于宽松，质量无法保证
- **85分**：质量与效率的平衡点

## 十、总结

### 10.1 两阶段工作流对比

| 维度 | 第一阶段 | 第二阶段 |
|------|----------|----------|
| **产出物** | 设定文档 | 章节正文 |
| **质量机制** | 正反辩论 | 预检-创作-审核-回流 |
| **迭代方式** | 设计者↔挑刺者 | 创作↔审核 |
| **计数器** | 辩论计数器 | 创作迭代计数器 |
