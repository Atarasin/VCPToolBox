## 一、设计背景与核心思想

### 1.1 V2版本的问题

| 问题 | 影响 |
|------|------|
| **假设一次完成所有详细设定** | 上下文超载，AI质量下降 |
| **没有上下文管理设计** | 新开窗口无法继续创作 |
| **没有按需展开机制** | 冗余设计，浪费token |
| **前期过度设计** | 很多详细设定后续用不上 |

### 1.2 V3核心设计思想

> **"骨架先行，血肉按需"**
> 
> 先建立不可变的基础框架，详细设定只在需要时展开

```
┌─────────────────────────────────────────────────────────────┐
│                    V3 核心设计原则                           │
├─────────────────────────────────────────────────────────────┤
│  1. 【渐进细化】从骨架到细节，按需展开，而非一次性完成        │
│  2. 【上下文可控】每个阶段只携带必要的最小上下文              │
│  3. 【状态外置】创作进度和关键信息持久化存储                  │
│  4. 【入口统一】任何阶段都有明确的入口条件和恢复机制          │
│  5. 【出口清晰】每个阶段完成后明确输出和下一跳条件            │
└─────────────────────────────────────────────────────────────┘
```

### 1.3 用户核心需求实现

| 需求 | V3解决方案 |
|------|-----------|
| 前期只设计最重要、不可变的内容 | **骨架层设计** - 世界观框架、势力分布、势力关系、重要人物基本信息 |
| 基于骨架设计小说大纲 | **大纲层设计** - 小说总纲 + 分卷大纲 |
| 分卷时确认登场势力/人物 | **登场标记机制** - 分卷大纲明确标注登场元素 |
| 按需展开详细设定 | **按需展开机制** - 根据登场标记触发详细设计 |
| 新开窗口能继续创作 | **上下文包管理** - 标准化"前情提要"组装规则 |

## 二、整体架构

### 2.1 五层架构设计

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           AI网络小说创作工作流 V3                              │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│  ┌──────────────┐    ┌──────────────┐    ┌──────────────┐    ┌────────────┐ │
│  │  Layer 0     │───▶│  Layer 1     │───▶│  Layer 2     │───▶│ Layer 3    │ │
│  │  需求层      │    │  骨架层      │    │  大纲层      │    │ 详细设定层 │ │
│  │              │    │  (不可变)    │    │              │    │ (按需展开) │ │
│  └──────────────┘    └──────────────┘    └──────────────┘    └─────┬──────┘ │
│       │                   │                   │                  │        │
│       ▼                   ▼                   ▼                  ▼        │
│  用户需求            核心框架            故事脉络            具体细节      │
│                                                                             │
│                              ┌──────────────┐                              │
│                              │  Layer 4     │                              │
│                              │  创作层      │                              │
│                              │  (正文输出)  │                              │
│                              └──────────────┘                              │
│                                                                             │
└─────────────────────────────────────────────────────────────────────────────┘
```

### 2.2 各层说明

| 层级 | 名称 | 内容 | 创建时机 | 大小控制 |
|------|------|------|----------|----------|
| Layer 0 | 需求层 | 创作需求文档(CRD) | 项目启动 | 500-1000字 |
| Layer 1 | 骨架层 | 世界观骨架 + 势力框架 + 人物框架 | 前期 | 3000-5000字 |
| Layer 2 | 大纲层 | 小说总纲 + 分卷大纲(含登场标记) | 骨架完成后 | 2000-4000字 |
| Layer 3 | 详细层 | 势力详情 + 人物详情 + 场景详情 | 按需展开 | 每份1000-2000字 |
| Layer 4 | 创作层 | 章节细纲 + 正文 | 逐章创作 | 每章3000-5000字 |

### 2.3 骨架层核心内容

#### 世界观骨架 (World Skeleton)

```yaml
世界观骨架:
  基础规则:
    - 修炼体系名称: "如：灵力体系"
    - 等级划分: "如：炼气-筑基-金丹-元婴-化神"
    - 核心资源: "如：灵石、丹药"
    - 世界层次: "如：凡界-灵界-仙界"
  
  地理框架:
    - 主要大陆/区域: "只列名称和基本定位"
    - 关键地点: "只列对剧情有决定性影响的地点"
  
  时代背景:
    - 当前时代: "如：灵气复苏后300年"
    - 重大历史事件: "只列影响当前格局的事件"
```

#### 势力骨架 (Faction Skeleton)

```yaml
势力骨架:
  顶级势力:
    - 势力名:
        定位: "如：正道魁首"
        核心特征: "一句话描述"
        立场: "正道/魔道/中立"
  
  势力关系矩阵:
    - "势力A - 势力B: 敌对/盟友/中立"
  
  重要人物索引:
    - 人物名:
        所属势力: ""
        身份: "如：宗主/天骄"
        立场标签: "如：正道领袖"
        与主角关系: "如：潜在盟友"
```

#### 主角骨架 (Protagonist Skeleton)

```yaml
主角骨架:
  基本信息:
    - 姓名:
    - 出身:
    - 初始身份:
  
  核心动机:
    - 表层目标: "如：为父报仇"
    - 深层目标: "如：打破阶级固化"
  
  核心特质:
    - 性格关键词: "如：坚韧、腹黑"
    - 能力特点: "如：剑道天赋"
  
  成长弧线:
    - 起点状态:
    - 终点状态:
    - 关键转变节点: "只标位置，不展开"
```

### 3.2 YAML元信息规范

所有文档必须包含以下YAML元信息：

```yaml
---
doc_type: "文档类型"              # world_skeleton/faction_frame/character_detail等
doc_id: "文档唯一标识"            # 如：world_main, sect_a, mc
version: "1.0"
created_at: "2026-03-21"
updated_at: "2026-03-21"
status: "draft"                  # draft/review/stable/deprecated

dependencies:
  required:                       # 必须依赖（创建前必须存在）
    - path: "10_skeleton/world/world_skeleton.md"
      reason: "需要世界观基础"
  optional: []                    # 可选依赖
  depended_by: []                 # 被哪些文档依赖

trigger: "创建触发原因"            # 详细文档需要标注触发原因
tags: ["标签列表"]
---
```

### 4.2 核心Agent详细设计

#### 挑刺者 ⭐V2继承的核心Agent

**角色定义**: 负责评审各类设定文档的质量，确保设定符合逻辑、一致且符合需求

**职责边界**:
- 评审骨架层设定：世界观骨架、势力框架、人物框架
- 评审大纲层设定：小说总纲、分卷大纲
- 评审详细层设定：势力详情、人物详情（按需展开后）
- **不**评审章节正文（由质量审核员负责）

**评审维度与权重**:

| 维度 | 检查内容 | 权重 |
|------|----------|------|
| 逻辑一致性 | 设定内部是否自洽 | 25% |
| 需求符合度 | 是否符合需求文档 | 20% |
| 创新性 | 是否有独特性 | 15% |
| 可扩展性 | 是否便于后续展开 | 15% |
| 完整性 | 必要信息是否齐全 | 15% |
| 表达清晰度 | 文档是否清晰易懂 | 10% |

**问题分级**:

| 级别 | 说明 | 处理要求 |
|------|------|----------|
| CRITICAL | 严重问题 | 必须修复 |
| MAJOR | 重要问题 | 最多允许1个 |
| MINOR | 轻微问题 | 建议修复 |
| SUGGESTION | 改进建议 | 可选采纳 |

**通过标准**:
- 总分 ≥ 85分
- 无CRITICAL问题
- MAJOR问题 ≤ 1个

**评审流程**:
```
设定文档完成
   │
   ▼
挑刺者评审
   │
   ├── 通过(≥85分) → 入库
   │
   └── 不通过(<85分) → 返回修改
                          │
                          │ (最多3轮)
                          ▼
                    超过3轮 → 转人工
```

## 九、V2 vs V3 对比

### 9.1 关键差异对比表

| 维度 | V2版本 | V3版本 |
|------|--------|--------|
| **设计时机** | 前期完成所有详细设定 | **骨架先行，详细设定按需展开** |
| **上下文管理** | 无明确设计 | **定义各阶段最小上下文包** |
| **前情提要** | 无 | **标准化前情提要模板** |
| **分层架构** | 模糊 | **清晰的5层架构** |
| **展开机制** | 无 | **明确的触发条件和决策流程** |
| **新开窗口** | 需要重喂所有内容 | **按清单组装上下文包** |
| **灵活性** | 低（设定固化早） | **高（延迟决策）** |
| **可维护性** | 低 | **高（分层清晰）** |
| **挑刺者角色** | 阶段1设定评审 | **保留并扩展至阶段1-3** |

### 9.2 V3核心改进点

1. **骨架层设计**: 明确区分"不可变基础"和"可变细节"
2. **按需展开**: 只在需要时才投入上下文资源设计详细设定
3. **上下文包标准化**: 每个阶段都有明确的"最小必要信息"清单
4. **前情提要机制**: 解决新开窗口的连续性问题
5. **分层递进**: 每层只依赖下层，降低耦合度
6. **保留挑刺者**: 继承V2的质量保障机制，评审标准保持不变（≥85分）

### 10.2 上下文包组装伪代码

```python
def assemble_context_package(stage, current_volume=None, current_chapter=None):
    """
    根据当前阶段组装上下文包
    """
    package = {
        "always_included": {
            "crd_summary": load("00_meta/project.yml"),
            "skeleton": {
                "world": load("10_skeleton/world/world_skeleton.md"),
                "faction": load("10_skeleton/factions/"),
                "protagonist": load("10_skeleton/characters/mc_frame.md")
            },
            "novel_outline": load("10_skeleton/plot/plot_skeleton.md")
        },
        "on_demand": {}
    }
    
    if stage in ["volume_design", "detail_expansion", "chapter_outline", "content_generation"]:
        package["on_demand"]["current_volume"] = load(f"30_content/volumes/vol_{current_volume:02d}/outline.md")
    
    if stage in ["detail_expansion", "chapter_outline", "content_generation"]:
        # 根据当前分卷大纲中的"需要展开的设定"清单加载
        expansion_list = get_expansion_list(current_volume)
        package["on_demand"]["faction_details"] = load_faction_details(expansion_list["factions"])
        package["on_demand"]["character_details"] = load_character_details(expansion_list["characters"])
    
    if stage in ["chapter_outline", "content_generation"]:
        package["on_demand"]["chapter_outline"] = load(f"30_content/volumes/vol_{current_volume:02d}/chapters/ch{current_chapter:03d}_outline.md")
    
    if stage == "content_generation":
        package["on_demand"]["previous_summary"] = generate_previous_summary(current_volume, current_chapter)
    
    return package
```
