## Problem Statement

The current MCP server in `modules/agentGateway` only exposes itself over local stdio. Remote workflows — cloud deployments, remote team members, or distributed clients — cannot connect to the Agent Gateway's MCP surface. The existing `mcpStdioServer.js` line-buffers JSON-RPC over `process.stdin/stdout`, which by definition only works when the MCP client and server share a local process boundary.

This means any setup where the Agent Gateway runs on a server and the MCP client (Claude Desktop, Cursor, a remote script, or another service) runs elsewhere is impossible without SSH tunnels or other fragile workarounds.

## What Makes This Cool

The "whoa" moment: your Agent Gateway becomes a **universal remote control** for AI agents. Right now `mcpBackendProxyAdapter.js` already speaks HTTP to a remote backend. If you add a WebSocket MCP transport, any remote client can connect to your gateway and get the full tool/prompt/resource surface without knowing anything about VCP's internal HTTP APIs. The MCP server becomes network-native.

The word you used was **"lightweight."** You want this to feel inevitable, not bolted-on. A thin `ws` upgrade on the existing Express server, wrapping the harness you already have. That's the aesthetic.

## Constraints

1. **Coexistence:** The existing stdio transport must continue working exactly as it does today. No regressions.
2. **Auth reuse:** Must use the existing gateway key / bearer token model from `GatewayBackendClient`, not invent a new auth scheme.
3. **Core immutability:** The harness business logic in `mcpBackendProxyAdapter.js` stays untouched. Only the transport layer (`mcpStdioServer.js`) changes.
4. **Stack alignment:** Use `ws` (already in `package.json` at `^8.17.0`) and the existing `./WebSocketServer.js` patterns.
5. **Protocol compliance:** Must speak JSON-RPC 2.0 over WebSocket frames, matching the existing `createBackendProxyMcpServerHarness` contract.

## Premises

1. **WebSocket is the right transport.** Bidirectional RPC is needed, and Express already exists. SSE would require two separate HTTP channels; HTTP polling is too chatty for JSON-RPC request/response correlation.
2. **Remote MCP should coexist with stdio.** Both transports call the same harness (`createBackendProxyMcpServerHarness`). Only the wire protocol changes.
3. **Auth should reuse the existing gateway auth model.** The `x-agent-gateway-key` / `Authorization: Bearer` model from `protocolGovernance.js` is sufficient. `resolveDedicatedGatewayAuth` checks `providedGatewayKey === config.gatewayKey`. During the WebSocket upgrade handshake, `request.headers` is available, so header-based auth works. Query parameters are a fallback for clients that cannot set headers.
4. **The MVP is a WebSocket endpoint on the existing Node server.** Wrap `createBackendProxyMcpServerHarness` in a thin transport layer. Validate with a single remote client.

## Approaches Considered

### Approach A: Thin WebSocket Wrapper (Minimal Viable)
- **Summary:** Add one file — `mcpWebSocketTransport.js` — that wraps `createMcpServerHarness` in a `ws` connection. Reuse the existing Express server for the upgrade handshake. No changes to `mcpAdapter.js`.
- **Effort:** Small (1 file, ~120 lines, 1 integration point in `server.js`)
- **Risk:** Low
- **Pros:** Ships in a day; zero risk to existing stdio transport; the harness logic is untouched.
- **Cons:** Shared harness instance means all connections see the same state; no batch request support; no server-initiated push (tools/listChanged stays false); auth is basic (query param or subprotocol).
- **Reuses:** `createBackendProxyMcpServerHarness`, existing `ws` module, Express upgrade handler pattern.

### Approach B: Transport Abstraction with Per-Session State (Recommended)
- **Summary:** Define a `McpTransport` interface. `StdioTransport` and `WebSocketTransport` both implement it. Each WS connection gets its own `requestContext` (isolated sessions). Support JSON-RPC batching. Heartbeat ping/pong. Server can push `tools/listChanged` and `resources/listChanged` notifications when capabilities update.
- **Effort:** Medium (3-4 new files, ~400 lines, refactor `mcpStdioServer.js` to use the new abstraction)
- **Risk:** Medium — touches existing stdio transport
- **Pros:** Stateless per connection; true bidirectional MCP (live updates); batching unlocks throughput; auth middleware at upgrade time; connection limits and graceful shutdown.
- **Cons:** Refactors working code; slightly more complex; needs careful testing to avoid regressing stdio.
- **Reuses:** `createBackendProxyMcpServerHarness` logic, `resolveDedicatedGatewayAuth` from `protocolGovernance.js`, existing `ws` module.

### Approach C: SDK-Based with Future-Proof Transport Layer
- **Summary:** Adopt `@modelcontextprotocol/sdk` as the server foundation. Build a custom `WebSocketServerTransport` that implements the SDK's transport interface, but bridges into the existing `createBackendProxyMcpServerHarness` for business logic.
- **Effort:** Medium-Large (2-3 new files + SDK dependency, ~300 lines, some adapter glue)
- **Risk:** Medium — introduces external dependency that may not match the auth model
- **Pros:** Aligns with official MCP SDK; future transport layers (SSE, HTTP) are near-free; battle-tested JSON-RPC framing.
- **Cons:** SDK auth is opinion-less — still need custom auth glue; SDK's `Server` class duplicates some of what `createMcpServerHarness` already does; adds a dependency for a problem solvable in ~100 lines.

## Recommended Approach

**Choose Approach B.**

Approach A leaves too much on the table (no sessions, no push, no batch). Approach C pays a dependency tax for marginal upside. Approach B gives you the architecture you want with a refactor that's contained to the transport layer — your core harness logic in `mcpBackendProxyAdapter.js` stays untouched.

6. **`./WebSocketServer.js`** (modified, ~+40 lines)
   - Add `/mcp` path handling **before** the existing `VCP_Key` regex checks and before the unhandled-path `socket.destroy()` call
   - The existing handler structure is: parse pathname → match `VCP_Key` regex routes → `socket.destroy()` for unmatched paths. `/mcp` becomes a new authenticated branch at the top of the matching chain

### Auth Design

During the WebSocket upgrade handshake, the `request` object has full HTTP headers. This means the existing header-based auth works at upgrade time.

**Primary auth (recommended):** Read `x-agent-gateway-key` or `Authorization: Bearer` from `request.headers` during upgrade, then validate using `resolveDedicatedGatewayAuth` from `modules/agentGateway/contracts/protocolGovernance.js`.

**Fallback auth:** For clients that cannot set custom headers (e.g., some browser WebSocket APIs), accept query parameters: `/mcp?gateway_key=xxx`. Bridge query params into the auth resolver by constructing a synthetic headers object:

```javascript
const syntheticHeaders = {
  [AGENT_GATEWAY_HEADERS.GATEWAY_KEY]: parsedUrl.query.gateway_key || '',
  authorization: parsedUrl.query.bearer_token ? `Bearer ${parsedUrl.query.bearer_token}` : ''
};
```

**Upgrade-time rejection:** If auth fails, call `socket.destroy()` immediately (matching the existing pattern in `WebSocketServer.js` for invalid `VCP_Key`). Do not complete the WebSocket handshake for unauthenticated requests.

### Per-Session State Model

The harness instance (`createBackendProxyMcpServerHarness`) remains a singleton, shared across all connections — just like the existing stdio transport's `runtimeState.context`. The singleton pattern is intentional because the `backendClient` and service bundle are expensive to recreate.

What is per-connection is the **`requestContext`** passed into each `handleRequest` call. The WebSocket transport injects a connection-scoped `sessionId` (e.g., `ws-${connectionId}`) into `params.requestContext` before calling `harness.handleRequest(request)`.

This means:
- **The harness is shared.** Both transports use the same singleton instance returned by `initializeBackendProxyMcpRuntime`. One instance, many connections.
- **`requestContext` is per-connection.** Memory search, context assembly, and job tracking are scoped to the connection's session.
- **`runtimeState` stays global.** `initializePromise` and `backendClient` are reused across stdio and WebSocket transports.

### Batch Requests

WebSocket transport enables JSON-RPC 2.0 batching because it is not line-buffered like stdio. `McpWebSocketTransport` handles batch requests:

```javascript
// Inside McpWebSocketTransport.onMessage:
if (Array.isArray(request)) {
  const responses = await Promise.all(
    request.map((r) => harness.handleRequest({
      ...r,
      requestContext: { ...r.requestContext, sessionId: this.connectionSessionId }
    }))
  );
  ws.send(JSON.stringify(responses.filter(Boolean)));
}
```

`McpStdioTransport` preserves the existing behavior: it rejects batch requests with code `-32600` (exactly as `mcpStdioServer.js` does today). Batch support is a WebSocket-only feature.

### Server-Initiated Push (Phase 2)

Once the transport layer is proven, enable `listChanged: true` in `buildMcpInitializeResult` and push notifications:

```javascript
// When capabilities change:
ws.send(JSON.stringify({
  jsonrpc: '2.0',
  method: 'notifications/tools/list_changed'
}));
```

This requires the capability service to emit events. Defer to a follow-up task.

## Open Questions

1. **Auth mechanism:** Primary = headers at upgrade time; fallback = query params. Confirmed feasible because `request.headers` is available in the `httpServer.on('upgrade')` event.
2. **Connection limits:** Default to 100 concurrent connections, configurable via `VCP_MCP_WS_MAX_CONNECTIONS`. Enforced in `WebSocketServer.js` before calling `wssInstance.handleUpgrade()`.
3. **`listChanged` trigger:** Should capability updates push immediately, or should the client poll? Defer push to Phase 2.
4. **CORS for WebSocket:** Do remote clients need CORS preflight for the upgrade handshake? Usually no for WS, but verify with the actual client stack.
5. **Binary vs text frames:** Stay text-only. MCP is JSON-RPC. No binary frames needed.
6. **Graceful shutdown:** Confirm that `WebSocketServer.js`'s existing `shutdown()` closes `wssInstance` and all client connections. `McpWebSocketTransport` connections are cleaned up automatically when `wssInstance` closes.

## Success Criteria

1. A remote client can open a WebSocket to `/mcp`, complete the MCP `initialize` handshake, and call `tools/list`.
2. The existing `npm run test:agent-gateway-mcp-transport` passes with zero regressions.
3. The existing stdio transport (`start:mcp-agent-gateway`) continues to work unchanged.
4. At least 100 concurrent WebSocket connections are supported without memory leaks.
5. JSON-RPC batch requests work over WebSocket.
6. Auth failure paths are tested: invalid `gateway_key`, missing auth, and rejected upgrade handshakes all return connection closure (not a completed WebSocket).
