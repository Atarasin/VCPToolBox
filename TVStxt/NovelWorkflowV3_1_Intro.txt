# AI长篇网络小说创作工作流 V3.1 核心指南

欢迎加入 V3.1 创作管线！本指南将帮助你快速理解当前工作流的核心思想、架构与运作机制。请在执行任务前仔细阅读，以确保你的输出符合整体规范。

---

## 核心设计思想：骨架先行，血肉按需

V3.1 版本的核心目标是**解决 AI 上下文超载问题，保障长篇创作的连贯性**。为此，我们确立了以下核心原则：

1. **渐进细化**：摒弃前期一次性完成所有设定的做法。从骨架到细节，按需展开。
2. **上下文可控**：每个阶段只携带完成任务所需的最小上下文包，绝不冗余。
3. **分层清晰**：明确区分“不可变的基础框架”与“可变的具体细节”。
4. **状态追踪**：通过明确的标记与状态文件，记录创作进度与触发条件。
5. **严苛审核**：所有设定必须经过挑刺者的严格审核才能生效，保证质量闭环。

---

## 五层架构总览

整个工作流被划分为五个层级（Layer），每个层级都有明确的输入、输出和负责的 Agent：

| 层级 | 名称 | 核心任务 | 负责 Agent | 产出物示例 |
|------|------|----------|-------------|------------|
| **Layer 0** | **需求层** | 明确创作需求与核心卖点 | 需求解析师（欧若拉） | `requirements.md` |
| **Layer 1** | **骨架层** | 搭建不可变的基础框架（极简！） | 世界观设计师（阿特拉斯）、人物设计师（普罗米修斯） | `world_skeleton.md`, 势力/人物 `_frame.md` |
| **Layer 2** | **大纲层** | 规划全书结构与单卷剧情 | 小说大纲规划者（代达罗斯）、分卷大纲规划者（克洛托） | `plot_skeleton.md`, `vol{N}/outline.md` |
| **Layer 3** | **详细层** | 根据剧情需要，按需展开设定细节 | 按需展开触发器、各详情设计师 | 势力/人物 `_detail.md` |
| **Layer 4** | **创作层** | 组装上下文，逐章进行正文创作 | 上下文包组装器、章节创作者、状态更新员 | `ch{NNN}.md`, 状态更新文档 |

*附注：**审判官（奥西里斯）** 贯穿 Layer 1-3，负责对所有设定进行逻辑挑刺与质量把控。*

## 核心流转机制

### 1. 骨架层的极简主义
在 Layer 1（骨架层），**绝对禁止过度展开细节**。
- 世界观只写基础规则和层级。
- 势力只写定位、特征和立场。
- 人物只写核心动机、性格特质和成长弧线锚点。
所有繁冗的设定（如宗门门规、具体城池布局、人物详细生平）必须留给 Layer 3。

### 2. 登场标记与按需展开
Layer 2 和 Layer 3 之间通过**「登场标记」**进行连接。
- **分卷大纲规划者（克洛托）** 在规划分卷大纲时，必须强制输出本卷涉及的势力与人物，并标记其状态（如 `[首次登场]`、`[回归]`、`[提及]`）。
- **按需展开触发器** 会读取这些标记。当发现有新势力/新人物登场且有重要戏份时，才会触发 Layer 3 的详细设定展开流程。

### 3. 上下文包管理
为了防止新开窗口导致上下文丢失或崩溃，Layer 4（创作层）引入了**上下文包组装器**。它会根据当前章节的需要，精准提取：
- **必带内容**：需求摘要 + 骨架层完整内容 + 小说总纲
- **按需内容**：当前分卷大纲 + 已展开的相关详细设定 + 前情提要（上章摘要、人物状态快照等）

### 4. 挑刺者审核闭环
从 Layer 1 到 Layer 3，任何 Agent 产出的设定文档，都必须通过 `ProposeWikiUpdate` 发起提案，并接受 **审判官（奥西里斯）** 的审核（`ReviewProposal`）。
- 审核分数必须 **≥85分**。
- 且不能包含任何 CRITICAL 级别问题。
- 且 MAJOR 级别问题 ≤ 1个。
如果被 Reject，原 Agent 必须无条件根据修改意见重新调整并再次提案。

## 你的行动准则

作为 V3.1 工作流中的一员，请时刻牢记：
- **认清你所在的层级**：不要越权执行其他层级的任务（比如在骨架层写细节，或者在大纲层写正文）。
- **遵守最小上下文原则**：只请求和输出当前任务必须的信息。
- **尊重审核机制**：认真对待奥西里斯的反馈，不要试图绕过审核。
- **规范输出**：严格遵守 Markdown 结构，确保其他 Agent 能够准确读取你的产出。
