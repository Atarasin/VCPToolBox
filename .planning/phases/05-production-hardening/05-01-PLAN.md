## Existing behavior to preserve

- `mcpWebSocketServer.js` already keeps `/mcp` isolated from the legacy websocket mesh and tracks live connections in a dedicated `Map`
- connection cleanup on `ws.close` and `ws.error` is already accepted behavior from Phase 2 and must remain intact
- the websocket manager already supports constructor-level `maxConnections` and `maxPayloadBytes` defaults, so implementation should tighten and expose these controls rather than replace them

## Gaps to close

- the documented environment-backed connection limit (`VCP_MCP_WS_MAX_CONNECTIONS`) is not yet wired through `server.js`
- there is still no timeout guard around stalled `/mcp` upgrade authentication (`WR-02`)
- payload ceilings do not yet have explicit endpoint coverage proving clean rejection and cleanup stability
- there is no phase-level regression check that rejected admissions or oversized payloads leave the manager in a healthy state for later connections
</context>

   - open and close multiple authenticated clients in sequence and in overlap
   - assert `getConnectionCount()` returns to zero after teardown

   - start a fixture with a very small `maxPayloadBytes`
   - send an oversized JSON frame
   - assert the client closes or errors promptly and the fixture can still shut down cleanly

   - after the oversized-frame client is rejected, connect a fresh healthy client and assert a normal request still succeeds

5. **Upgrade auth timeout**
   - inject a deterministic auth-stall or delayed-upgrade fixture path

Testing constraints:
- use explicit per-wait timeouts for close/error assertions
- keep leftover socket tracking in the fixture so `server.close()` cannot hang
- do not rely on brittle timing-only assertions when a direct connection-count or close-event assertion is available
