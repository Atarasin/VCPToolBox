## Existing behavior to preserve

- inbound messages are serialized per websocket connection today, which already prevents concurrent queue execution on one socket
- `maxBatchSize` continues to bound JSON-RPC fan-out within a single websocket frame
- heartbeat ping/pong is transport-owned and must not be counted as application abuse traffic

## Gaps to close

- there is no per-connection message rate limiter before `handleClientMessage()` dispatches backend work
- there is no documented remote behavior for transport-level throttling, retry timing, or burst recovery
- there is no websocket regression coverage proving that one abusive client cannot impact a separate healthy client connection
</context>

  <action>
Add a per-connection limiter in the websocket transport layer so excessive inbound traffic is rejected before shared backend work is dispatched.

Required implementation choices:
- keep the limiter state per authenticated connection, not global across all clients
- choose one explicit counting model and document it in code:
  - preferred default: count each inbound websocket application frame as one message, while relying on `maxBatchSize` to bound per-frame amplification
  - if you choose weighted batch-member counting instead, document why and keep the tests aligned with that rule
- add configurable limit settings using the existing `VCP_MCP_WS_*` naming family if the implementation needs runtime tuning
- when the limiter rejects a request that expects a response, return a stable JSON-RPC error payload with actionable metadata such as a stable code and `retryAfterMs`
- when the limiter rejects a notification-only message, prefer a transport-safe behavior that avoids backend work and remains consistent with JSON-RPC expectations

Design constraints:
- do not route transport-local rate limiting through the backend-proxy adapter just to reuse existing tool-level operability helpers
- do not count server-originated ping/pong traffic against the client limit
- do not allow rejected messages to keep accumulating unbounded state in the serialized queue

Extend the websocket suite with focused endpoint tests for transport-level rate limiting.

Required test cases:
1. **Burst rejection on one connection**
   - configure a very small per-connection rate limit
   - send a short burst of valid JSON-RPC requests on one websocket
   - assert early requests succeed and the overflow request receives the documented rate-limit behavior

2. **Retry metadata**
   - assert the rate-limited response includes stable metadata such as the selected code, `retryAfterMs`, and any configured limit/window values the implementation exposes

3. **Window recovery**
   - after waiting past the configured window, send another request on the same websocket
   - assert the connection recovers without reconnecting

4. **Healthy-peer isolation**
   - keep one client in an abusive burst pattern while a second authenticated client sends a normal request
   - assert the second client still receives a normal response and is not throttled by the first client

Optional if needed for the chosen design:
- add a batch-specific rate-limit test if batch messages are counted with weighted cost rather than one frame

Testing constraints:
- keep burst sizes small and deterministic so the test remains stable under CI timing jitter
- use explicit per-wait timeouts for all expected responses
- if the limiter closes abusive sockets instead of only returning errors, assert that this is intentional and stable rather than incidental
