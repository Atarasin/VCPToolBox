## Existing behavior to preserve

- `modules/agentGateway/mcpWebSocketServer.js` already parses one JSON object per inbound WebSocket text frame
- parse failures already return shared `-32700` JSON-RPC envelopes
- single non-object payloads already return shared `-32600` invalid-request envelopes
- `modules/agentGateway/transport/webSocketTransport.js` is a dumb pipe and should stay free of JSON-RPC semantics
- `modules/agentGateway/mcpStdioServer.js` must continue to reject JSON-RPC batch arrays

## Gap to close

- WebSocket arrays are still rejected even though Phase 3 requires bounded batch support
- There is no configurable maximum batch size yet
- Current websocket tests only verify batch rejection, not valid batch aggregation rules

  <action>
Refactor the request-dispatch path in `modules/agentGateway/mcpWebSocketServer.js` so one WebSocket text frame can carry either a single JSON-RPC request object or a bounded JSON-RPC batch array.

Implementation requirements:
- Keep the current single-request path intact for non-array JSON payloads.
- Add an explicit maximum batch size setting on the websocket manager, configurable through an option and/or an MCP WebSocket-specific environment variable. Pick one canonical name and use it consistently.
- Reject an empty batch with JSON-RPC `-32600`.
- Reject a batch larger than the configured limit with JSON-RPC `-32600` and machine-readable details that identify the limit.
- For valid batches:
  - process each entry through the same request-context injection path used for single requests
  - preserve array order for response items
  - omit notification entries that do not expect responses
  - if every item is a notification, send no response frame
- For invalid batch members, return a per-entry invalid-request error object with `id: null` rather than crashing the whole connection.
- Keep transport logging and internal-error shaping aligned with the current single-request implementation.
- Do not add batch support to stdio.

Recommended approach:
- Extract a helper that handles one normalized request item and returns either a JSON-RPC response object or `null`
- Extract a batch helper that loops over array items, awaits each item predictably, and returns either `null`, a single error object, or a response array depending on the JSON-RPC case

Required test cases:

   - Send a two-item batch of ordinary requests
   - Assert the server returns one response frame containing an array with responses in the original request order

2. **Mixed request + notification batch**
   - Include at least one item without `id`
   - Assert notification entries produce no response element while request entries still appear

3. **All-notification batch**
   - Send only notification items
   - Assert the websocket receives no response frame within a short timeout and the connection stays healthy

4. **Empty batch rejection**
   - Send `[]`
   - Assert a `-32600` invalid-request response

5. **Oversized batch rejection**
   - Configure a small test-only maximum batch size
   - Send a larger batch and assert the response identifies the configured limit

6. **Invalid member shaping**
   - Include a batch member that is not a plain JSON-RPC object
   - Assert the response array includes an `id: null` invalid-request error for that entry while valid request entries still complete

Keep the new tests websocket-specific. Avoid duplicating stdio transport cases that are already covered elsewhere.
