## Existing behavior to preserve

- `createMcpError()` plus harness try/catch already produce top-level JSON-RPC errors for malformed or unsupported prompt/tool/resource requests
- `createFailureResult()` already produces MCP tool results with `isError: true` for gateway-managed tool runtime/policy failures
- `mapGatewayFailureToMcpErrorCode()` already translates canonical Gateway failures into `MCP_*` codes

## Gaps to close

- websocket tests do not yet prove the remote error contract for prompts, tools, and resources end to end
- there is no websocket-first proof that clients are guided to `prompts/get` when they misuse `gateway_agent_render` as a tool
- there is no websocket-first proof that resource errors use MCP-standard codes without exposing raw stack traces
- unexpected backend/network exceptions can still bubble through the harness with raw `error.message`, so the plan must explicitly audit connection-detail leakage as part of CAP-05
</context>
