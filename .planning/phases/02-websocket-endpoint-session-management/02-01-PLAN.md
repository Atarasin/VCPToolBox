## Phase 1 layering to preserve

- `modules/agentGateway/transport/stdioTransport.js` defines the callback-based dumb-pipe contract
- `modules/agentGateway/transport/mcpTransport.js` currently omits the de-facto `finished` property and must be updated before adding a second transport implementation
- `modules/agentGateway/mcpStdioServer.js` proves transport code should not own JSON-RPC semantics
- `modules/agentGateway/adapters/mcpBackendProxyAdapter.js` expects `handleRequest(request)`, not `handleRequest(request, context)`

## Phase 2 implementation constraints

- `/mcp` must stay isolated from `WebSocketServer.js`
- Upgrade auth must reuse `resolveDedicatedGatewayAuth()`
- Canonical `sessionId` is server-generated once per connection
- Request metadata injection must happen before `harness.handleRequest()`
- Keepalive must use native `ws` ping/pong, not JSON heartbeat messages

## Existing project patterns to reuse

- Shared HTTP server + `noServer` upgrade flow
- Immediate `socket.destroy()` for rejected websocket paths
- Dedicated `Map` registries for active websocket clients
- Small helper modules under `modules/agentGateway/`

Create `modules/agentGateway/mcpWebSocketServer.js` that owns the full `/mcp` websocket lifecycle while preserving the existing MCP harness contract.

Required exports and shape:
- Export `createMcpWebSocketServer(options = {})`.
- `createMcpWebSocketServer(options)` accepts optional `options.harness`; when omitted it lazily creates the harness using the same backend-proxy runtime initialization path as the stdio server.
- The returned manager must expose lifecycle methods that are usable from `server.js`: at minimum `attach(httpServer)` or `initialize(httpServer)`, plus `close()`.
- Expose a lightweight diagnostics helper such as `getConnectionCount()` for tests and shutdown assertions.

Implementation requirements:
1. **Dedicated `ws` server**
   - Use `new WebSocket.Server({ noServer: true, clientTracking: false })`.
   - Do not add `/mcp` handling to `WebSocketServer.js`.

2. **Upgrade auth**
   - In the shared HTTP server `upgrade` flow, return early unless the pathname is exactly `/mcp`.
   - Authenticate with `resolveDedicatedGatewayAuth({ headers: request.headers, pluginManager })`.
   - On missing/invalid auth, call `socket.destroy()` before the websocket handshake completes.
   - Preserve authenticated `gatewayId` when present.

3. **Connection-scoped identity**
   - Generate a dedicated connection ID and canonical `sessionId` server-side using a collision-resistant built-in source such as `crypto.randomUUID()`.
   - Store each live connection in a dedicated `Map`.
   - Build a trusted base context per connection: `sessionId`, `runtime`, `source`, optional `gatewayId`, and any other non-secret diagnostic fields you need.

4. **Harness dispatch**
   - Reuse the stdio-server pattern: transport is a dumb pipe, manager parses JSON and calls `harness.handleRequest(request)`.
   - Reuse `createJsonRpcErrorResponse` from `mcpStdioServer.js` so websocket and stdio error envelopes stay byte-for-byte aligned where applicable.
   - Reject JSON-RPC batch requests explicitly with `-32600` rather than deferring behavior implicitly.
   - Before dispatch, merge inbound request params with a canonical connection-scoped context:
     - preserve client `requestId` and `agentId` when valid
     - generate `requestId` when missing via `normalizeRequestContext()`
     - always overwrite `sessionId` with the server-generated canonical value
     - include authenticated `gatewayId` in `requestContext`, and mirror it into `authContext` if needed for downstream compatibility
   - Use `normalizeRequestContext()` rather than `resolveNativeRequestContext()` because websocket requests are already structured JSON-RPC payloads after upgrade.
   - Keep request handling sequential per connection using a promise queue, matching the Phase 1 ordering pattern.

5. **Keepalive and cleanup**
   - Implement native heartbeat with websocket `ping()` / `pong`.
   - Maintain `isAlive` state per connection; terminate stale clients after the chosen interval strategy.
   - Cleanup must be idempotent and run on `close`, `error`, and forced termination:
     - remove the connection from the dedicated `Map`
     - clear heartbeat timers
     - close/finish the transport without double-closing

6. **Error behavior**
   - Reuse `createJsonRpcErrorResponse` from `mcpStdioServer.js` where applicable.
   - Do not leak auth failure detail to the websocket client on rejected upgrades.
   - Log server-side failures consistently with the Agent Gateway MCP transport prefix pattern.

Keep the implementation self-contained under `modules/agentGateway/` and add small helpers inside the file only if they materially improve readability.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP upgrade -> websocket manager | Untrusted client headers and requested path cross into authenticated websocket establishment |
| WebSocket frame -> MCP manager | Untrusted inbound text frames become parsed MCP requests |
| Connection context -> harness | Manager-injected session/auth metadata becomes authoritative for downstream job visibility |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Spoofing | Upgrade auth | mitigate | Reuse `resolveDedicatedGatewayAuth()` and reject unauthorized upgrades with `socket.destroy()` |
| T-02-02 | Tampering | `sessionId` injection | mitigate | Always overwrite canonical `sessionId` from the connection context before dispatch |
| T-02-03 | Elevation of Privilege | Endpoint isolation | mitigate | Keep `/mcp` in a dedicated manager with a private connection `Map` and no shared legacy routing |
| T-02-04 | Denial of Service | Stale websocket connections and malformed/binary input | mitigate | Native ping/pong keepalive, binary-frame drop, and deterministic timer cleanup |
| T-02-05 | Resource Leakage | Connection lifecycle | mitigate | Centralized idempotent cleanup on `close`, `error`, and termination |
</threat_model>
