## Existing startup/shutdown integration points

From `server.js`:
- `pluginManager.setWebSocketServer(webSocketServer)` happens before service initialization and must remain unchanged for the legacy mesh
- after `app.listen(...)`, the current server boot path initializes `webSocketServer.initialize(server, ...)`
- `gracefulShutdown()` already calls `webSocketServer.shutdown()` and then plugin/KB cleanup; the new `/mcp` manager must close before plugin teardown and should not race auth/config access during shutdown

Create `test/agent-gateway/adapters/agent-gateway-mcp-websocket.test.js` and add a dedicated npm script in `package.json`.

Test design requirements:
- Use `node:test`, `node:assert/strict`, and real `ws` client/server connections.
- Reuse the current MCP fixture/runtime pattern where practical so stdio and websocket tests stay structurally similar.
- Attach `createMcpWebSocketServer()` to a lightweight in-process HTTP server rather than booting the full production app.

Required test cases:
1. **Authenticated connect via gateway key header**
   - Connect to `/mcp` with the dedicated gateway header.
   - Assert the websocket opens successfully and can exchange a representative MCP request/response.

2. **Authenticated connect via bearer token**
   - Connect with `Authorization: Bearer ...`.
   - Assert the connection is accepted when the bearer token matches the configured gateway auth contract.

3. **Unauthorized upgrade rejected before open**
   - Attempt connection without valid credentials.
   - Assert the client never reaches open state and the server does not register a live connection.

4. **Malformed JSON parse error**
   - Send invalid JSON over an authenticated websocket.
   - Assert the manager returns JSON-RPC parse error `-32700` using the shared error envelope.

5. **Batch request rejection**
   - Send a JSON array over an authenticated websocket.
   - Assert the manager rejects it with JSON-RPC invalid request error `-32600`.

6. **Canonical session continuity across requests**
   - Send multiple MCP requests over the same websocket.
   - Use a stub or instrumented harness to capture inbound request payloads.
   - Assert every dispatched request sees the same server-generated canonical `sessionId`, even if the client tries to send a different one.
   - Assert `requestId` is generated when missing and preserved when already valid.
   - Assert authenticated `gatewayId` is propagated into `requestContext` and any required compatibility auth fields.

7. **Cleanup removes connection state on close/error**
   - Close the client or force a transport error.
   - Assert the manager's connection count returns to zero and no timers keep the test process alive.
   - Cover timer cleanup specifically on the websocket error path.

8. **Native keepalive path**
   - Exercise the heartbeat loop with short test-only intervals.
   - Assert healthy clients remain connected when they respond to ping/pong, and stale clients are terminated/cleaned up when they do not.

9. **Isolation from the legacy mesh**
   - Prove `/mcp` routing uses the dedicated manager path rather than `WebSocketServer.js` state or auth flow.
   - This can be asserted indirectly by using a dedicated manager fixture without legacy mesh initialization plus a code-path assertion where useful.

10. **Shutdown cleanup**
   - Close the attached HTTP server / manager fixture in the same order planned for `server.js`.
   - Assert connection state and timers are released so the test process exits cleanly.

Add `package.json` script:
- `"test:agent-gateway-mcp-websocket": "node --test test/agent-gateway/adapters/agent-gateway-mcp-websocket.test.js"`

Keep the new tests focused on websocket endpoint behavior. Do not duplicate the stdio transport test matrix unless a websocket-specific invariant is being checked.
