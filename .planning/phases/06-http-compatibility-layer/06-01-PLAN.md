## Existing behavior to preserve

- `/mcp` websocket upgrade handling in `mcpWebSocketServer.js` remains untouched and continues to own the upgrade path
- `initializeBackendProxyMcpRuntime()` stays the only runtime bootstrap path for transport-owned MCP servers
- the backend-proxy harness remains the canonical owner of MCP lifecycle, prompt, tool, and error semantics

## Gaps to close

- there is no HTTP MCP transport for Trae or other HTTP-first clients today
- there is no server-owned `MCP-Session-Id` lifecycle for HTTP MCP follow-up requests
- there is no parity test matrix proving HTTP MCP matches stdio/WebSocket behavior for representative lifecycle and capability methods
</context>

  <action>
Implement a new HTTP MCP transport that follows the same architectural boundary already accepted for stdio and WebSocket.

Required implementation scope:
- add a new module (for example `mcpHttpServer.js`) that:
  - reuses `initializeBackendProxyMcpRuntime()`
  - keeps a transport-local session map keyed by `MCP-Session-Id`
  - creates a new session on `initialize`
  - rejects follow-up requests with missing or unknown session IDs unless the request itself is a fresh `initialize`
  - rejects the next `initialize` when the active-session limit is reached instead of evicting an existing session
  - expires idle sessions and frees all related server-side state
  - injects canonical request/session/auth context before delegating to `harness.handleRequest()`
- enforce dedicated auth using the same gateway-key / bearer-token rules already used by the websocket transport
- support the canonical Streamable HTTP request flow on `/mcp` using Express `POST`, required `GET`, and `DELETE`
- own route registration inside the transport adapter and mount it from `server.js`; do not spread HTTP MCP bookkeeping into unrelated route files
- mount a route-local JSON body parser whose limit is controlled by `VCP_MCP_HTTP_MAX_PAYLOAD_BYTES`
- mirror the existing websocket hardening defaults with:
  - `VCP_MCP_HTTP_MAX_SESSIONS` (default 100)
  - `VCP_MCP_HTTP_MAX_PAYLOAD_BYTES` (default 4 MB)
  - `VCP_MCP_HTTP_RATE_LIMIT_MESSAGES` (default 60)
  - `VCP_MCP_HTTP_RATE_LIMIT_WINDOW_MS` (default 1000)
  - `VCP_MCP_HTTP_AUTH_TIMEOUT_MS` (default 5000)
  - `VCP_MCP_HTTP_SESSION_IDLE_MS` (default 600000)
- require `GET /mcp` as the server-to-client SSE stream, validate `MCP-Session-Id`, emit heartbeat frames, and keep resumable replay out of scope
- support `DELETE /mcp` to abort pending work and remove the session record immediately
- propagate an `AbortSignal` into every `harness.handleRequest()` call and abort on `req.aborted`, session DELETE, or idle expiry
- on GET stream backpressure, await `drain` for JSON-RPC response frames and prefer dropping stale heartbeat frames over blocking forward progress
- mount the HTTP handler so it coexists with the existing `/mcp` websocket upgrade flow on the same `http.Server`

Design constraints:
- do not duplicate MCP capability logic in route handlers
- do not move mutable session state into the backend-proxy harness
- do not modify `mcpStdioServer.js`, `mcpWebSocketServer.js`, or `createBackendProxyMcpServerHarness()` to make HTTP work
- keep comments short and only where HTTP session or coexistence logic is non-obvious

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Remote HTTP client -> `/mcp` POST route | Authenticated requests must not gain implicit cross-request state without server-owned session validation |
| Remote HTTP client -> `/mcp` GET/DELETE routes | Streaming and teardown must enforce the same session ownership and auth boundary as POST |
| HTTP transport session map -> shared MCP harness | Mutable transport session state must stay out of the shared business runtime |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Spoofing | Anonymous follow-up HTTP MCP request | mitigate | Require valid `MCP-Session-Id` after initialize |
| T-06-02 | Elevation of Privilege | HTTP transport weakens dedicated auth vs websocket | mitigate | Reuse `resolveDedicatedGatewayAuth` semantics |
| T-06-03 | Reliability | `/mcp` HTTP handler collides with websocket upgrade behavior | mitigate | Keep request and upgrade paths separate and verify coexistence |
| T-06-04 | Tampering | Route layer mutates MCP business semantics | mitigate | Delegate to the shared backend-proxy harness after transport shaping only |
| T-06-05 | Denial of Service | HTTP path misses websocket hardening parity | mitigate | Mirror session, payload, rate-limit, timeout, and idle-expiry controls with explicit env knobs |
| T-06-06 | Reliability | Slow GET stream consumer blocks transport progress | mitigate | Await `drain` for response frames and drop stale heartbeat frames when necessary |
</threat_model>
